## 1. What already exists in this repo (read this before writing anything)

This repo is auto-synced by the **LeetHub v2** Chrome extension. Every time a problem is solved on LeetCode, LeetHub commits a new folder straight to `master` on GitHub. That sync is already fully automatic and already working — **do not build any scraping, syncing, or "fetch from LeetCode" logic.** The web app only ever needs to read what's already sitting in this repo.

Structure at the repo root:

```
/0002-add-two-numbers/
    0002-add-two-numbers.<ext>      # the solution code, extension = language
    README.md                        # raw HTML problem statement + difficulty
/0015-3sum/
    0015-3sum.rkt
    README.md
... (176 of these folders today, more added over time)
README.md         # repo-level README, see section 1.3
stats.json         # see section 1.2
```

### 1.1 Per-problem `README.md`
Contains the LeetCode problem statement as raw HTML (title as `<h2><a href="...">N. Title</a></h2>`, difficulty as `<h3>Easy|Medium|Hard</h3>`, then description, examples, constraints as HTML). Parse this for the problem title, LeetCode link, difficulty, and description — don't just dump raw HTML into the page, sanitize/render it properly.

### 1.2 `stats.json`
Shape:
```json
{
  "leetcode": {
    "easy": 109, "medium": 33, "hard": 32,
    "shas": {
      "0509-fibonacci-number": {
        "0509-fibonacci-number.cpp": "<sha>",
        "README.md": "<sha>",
        "difficulty": "easy"
      },
      "README.md": { "": "<sha>" },
      "stats.json": { "": "<sha>" },
      ...
    }
  }
}
```
Important details:
- Skip the `"README.md"` and `"stats.json"` keys inside `shas` — they aren't problems.
- Every other key is a problem folder name, and gives you its `difficulty` directly (cheaper than parsing HTML for this).
- **The insertion order of keys in `shas` is the chronological solve order** (oldest first). Use this to power a "recently solved" feed — no need to shell out to `git log`.
- The non-`difficulty` key in each entry (e.g. `"0509-fibonacci-number.cpp"`) tells you the exact code filename — use its extension to detect the language.

### 1.3 Root `README.md`
Line 1–3 is hand-written intro text. From `<!---LeetCode Topics Start-->` to `<!---LeetCode Topics End-->` is **auto-regenerated by LeetHub on every push** — it's a big list of `## Category Name` headers each followed by a markdown table of links to every problem folder tagged with that category. This is the **only source of category/topic tags** — there's no per-problem category field anywhere else, so parse this block to build a `slug -> [categories]` map.

Categories currently present (42): Math, Dynamic Programming, Recursion, Memoization, Database, String, Tree, Depth-First Search, Breadth-First Search, Binary Search Tree, Binary Tree, Two Pointers, Array, Greedy, Sorting, Counting Sort, Matrix, Simulation, String Matching, Hash Function, Hash Table, Stack, Sliding Window, Bit Manipulation, Design, Heap (Priority Queue), Data Stream, Linked List, Prefix Sum, Geometry, Graph Theory, Shortest Path, Ordered Set, Enumeration, Number Theory, Union-Find, Queue, Topological Sort, Game Theory, Combinatorics, Bitmask, Monotonic Stack, Backtracking.

**⚠️ Critical constraint: never hand-edit anything between the Start/End markers, and never let the web app's build output touch the root README.md at all.** LeetHub owns that block; if the app's build script and LeetHub fight over the same file you'll get merge conflicts on every solve.

## 2. Where the web app code lives

Put the entire Next.js project in a **new subfolder**, e.g. `/webapp`, at the repo root — **not** mixed into the root alongside the problem folders. Reasons:
- Keeps `node_modules`, `package.json`, `.next/` etc. out of the way of the 176+ problem folders and whatever LeetHub commits next.
- Lets you set Vercel's **Root Directory** setting to `webapp` so Vercel only builds that subtree.
- Means the data-generation script can safely treat "everything in the repo root except `webapp/`, `.github/`, `README.md`, `stats.json`" as a problem folder, with zero risk of the app accidentally including itself as a "problem."

```
/webapp/
  app/                  # Next.js App Router
  components/
  lib/
    parse-repo.ts       # build-time script, see section 3
  public/
  package.json
  ...
/0002-add-two-numbers/   # untouched, LeetHub keeps managing these
...
README.md                 # untouched except intro section, see section 6
stats.json
```

## 3. Data pipeline (build time, not runtime)

Because the app lives in the same repo as the data, **read the data straight off the filesystem at build time** — do not call the GitHub API at runtime. This means:
- No API tokens, no rate limits, no runtime fetch latency.
- Every new deploy (triggered by any push — see section 5) re-scans the current filesystem, so new problems appear automatically on the next build.

Write one build-time script (e.g. `lib/parse-repo.ts`, run via a `prebuild` npm script or directly inside `generateStaticParams`/build-time code) that:

1. Walks the repo root (one level up from `webapp/`), collecting every directory matching `^\d{4}-[a-z0-9-]+$`.
2. For each folder:
   - Reads `stats.json` to get `difficulty` and the code filename (fallback: just glob the folder for the one non-`README.md` file if a folder is ever missing from `stats.json`).
   - Reads the code file, detects language from extension (map: `.py`→Python, `.java`→Java, `.ts`→TypeScript, `.js`→JavaScript, `.ex`→Elixir, `.erl`→Erlang, `.sql`→SQL, `.rkt`→Racket, `.cpp`→C++), and keeps the full source for a code preview.
   - Reads `README.md`, extracts problem number + title (from the `<h2><a>` tag), the LeetCode URL (from the `href`), and the description HTML (sanitize before rendering — see section 4).
   - Looks up this folder's categories from the parsed root `README.md` topics block (section 1.3).
   - Records its position in `stats.json`'s key order → a `solvedIndex` used for "recently solved" sorting.
3. Emits a single JSON artifact (e.g. `webapp/lib/generated/problems.json`, git-ignored, regenerated every build) that the pages consume. Don't hit the filesystem from page components directly — go through this one generated artifact so every page shares one consistent shape.

Data shape per problem (adjust as needed, but cover all of this):
```ts
{
  slug: string;            // "0015-3sum"
  number: number;          // 15
  title: string;           // "3Sum"
  leetcodeUrl: string;
  githubUrl: string;       // link back to the folder in this repo
  difficulty: "Easy" | "Medium" | "Hard";
  categories: string[];
  language: string;        // "Racket"
  languageExt: string;     // "rkt"
  code: string;             // full source, for the preview + detail page
  descriptionHtml: string;  // sanitized
  solvedIndex: number;      // position in stats.json solve order
}
```

## 5. Deployment on Vercel — and how the "daily auto-update" actually works

You asked for the site to update automatically when you solve a new problem. Here's the important part: **you don't need to build any cron job, scheduler, or polling logic for this.** Vercel's Git integration already redeploys automatically on every push to the connected branch — and LeetHub already pushes a commit to this repo every time you solve a problem. Chain those two together and it's not just daily, it's near-real-time:

1. Create a new Vercel project, import this GitHub repo.
2. In Project Settings → **Root Directory**, set it to `webapp`.
3. Leave the default Build/Install commands (Vercel auto-detects Next.js).
4. That's it. From then on:
   - You solve a problem on LeetCode → LeetHub commits the new folder + updates `stats.json` and the root `README.md` topics block → GitHub push → **Vercel's existing webhook fires automatically** → new build runs → the build-time script in section 3 re-scans the filesystem, sees the new folder → new deploy goes live with the new problem included.
5. No environment variables are needed for this pipeline since everything is read from the filesystem at build time, not fetched over the network.
6. Optional hardening: add a `vercel.json` ignoring builds if only non-`webapp` files that don't affect it changed — not necessary, but if solve frequency is very high and you want to save build minutes, you can restrict triggers. Not required for correctness.

If Antigravity or you later decide to host the web app in a **separate** repo instead of this one, this whole section changes (you'd need the GitHub API + a webhook or scheduled revalidation) — but per your choice, same-repo is simpler and preferred here.

## 6. Visual design direction

Style: **colorful and playful**, not a plain dev-portfolio grid.

- Difficulty color coding used consistently everywhere (badges, chart segments, card left-border accents): Easy = green, Medium = amber/orange, Hard = red/pink.
- A distinct accent color per major category group is a nice touch (e.g. warm tones for math/DP, cool tones for trees/graphs) but keep it systematic, not random per card.
- Friendly rounded display font for headings paired with a clean readable body font; a monospace font for code.
- Small playful touches: hover-tilt or lift on cards, animated count-up numbers on the stats dashboard, a little emoji per category in chip labels (e.g. 🧮 Math, 🌳 Tree, 🔍 DFS, 🧩 Backtracking), soft gradient backgrounds rather than flat white/gray.
- Keep it fast and legible — playful doesn't mean cluttered. Load the frontend-design guidance available in the IDE/toolchain before styling, and keep contrast/readability solid on code blocks.
